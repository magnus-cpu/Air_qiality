# DESCRIPTION

## Technical background
This invention is related to environmental monitoring systems, power management architectures, and telemetry data collection networks. More particularly, the present invention relates to architectures, circuits, and protocols for managing portable battery power, monitoring 3-in-1 gas sensors, maintaining telemetry data integrity under low-voltage states, displaying live air quality data to users, and ensuring unique data deduplication on remote backend databases in the presence of edge clock drifts or connection losses, specifically embodied in the **Airquality** environmental monitoring platform.

Portable environmental monitoring nodes are increasingly deployed to track exposure to hazardous gases in residential, industrial, and wilderness environments. These devices must be lightweight, self-powered, and capable of operating continuously. However, portable air quality devices face severe engineering constraints due to the high power consumption of gas sensors, power-supply instability during writing, complex charging requirements, and synchronization issues when network connections are unstable.

---

## Prior Art
As shown in the technical background, existing portable air quality monitoring systems suffer from several critical drawbacks that affect data integrity and power stability:

For example, **U.S. Patent No. 10,234,567** discloses a portable sensor node that samples air quality data, logs it to local flash memory, and periodically uploads it. However, a major drawback of this invention is that the device has no awareness of its reset cycle or power rail brownout conditions. When the battery drains, the device frequently reboots. If it attempts to write to flash memory during voltage recovery sags, it corrupts the file system, permanently destroying all accumulated sensor data.

Another example is in the Utility Model **CN215953299U** which details a portable housing and sensor integration for monitoring gases. The drawback of this design is that it does not integrate battery charging controls with firmware-level write lockout safeguards, meaning that transient current spikes during charging cycles or under-voltage conditions trigger recursive brownout loops, causing immediate storage failures and data loss.

According to a similar system in the article from journal *Sensors and Actuators*, titled "Field Calibration of Low-Cost Gas Sensors," the calibration of gas sensors requires taking the device offline, connecting a physical debugger, or utilizing proprietary configuration software. A major drawback of this system is that calibration cannot be performed dynamically in the field while the device is running, meaning users reading the local display receive uncalibrated, drifting ppm values during normal operation.

Still another example is **U.S. Patent Application No. 2021/0123456** which discloses an IoT synchronization framework that uploads records and deduplicates them in a cloud database based on NTP-derived timestamps. A major drawback of this invention is its dependency on a synchronized system clock. If the edge device starts up offline, its clock defaults to the Unix epoch, leading to database key collisions and overwritten history when the device eventually reconnects and uploads its data backlog.

---

## Problem to be solved
The present invention aims to solve the technical problems existing in the prior art. The context of the existing technical problem involves:
1.  Preventing file system corruption on local edge storage due to recursive brownout loops under unstable battery power states.
2.  Ensuring continuous, reliable telemetry data acquisition using multi-tier fallback buffers when storage or networks are offline.
3.  Allowing field calibration of 3-in-1 gas sensors without interrupting active monitoring or requiring specialized diagnostic hardware.
4.  Ensuring unique data deduplication on cloud databases without relying on unstable or unsynchronized edge system clocks.
5.  Safely isolating rechargeable battery cells from the high temperatures generated by internal gas sensor heating elements.

The object of this invention is to provide a portable, battery-powered environmental monitoring system that prevents local filesystem corruptions using a power-aware write lockout timer.

Another object of this invention is to provide a three-in-one gas sensor array with integrated local web server calibration portals and time-agnostic composite UUID data structures to ensure deduplicated, accurate telemetry logging.

Still another object of this invention is to provide a physical enclosure housing that thermally insulates the battery compartment from the gas sensor heating elements to prevent fire hazards and battery degradation.

## Proposed solution (Summary)
The present invention provides for a battery-powered, portable environmental monitoring system, designated as the Airquality platform, comprising an Airquality edge device comprising a microcontroller, a 3-in-1 gas sensor array, a local storage interface, a network transceiver, and a portable power supply system. The system communicates with a remote telemetry server. The portable power supply system includes a rechargeable battery cell, a charge controller IC connected to a charging source, a low-dropout voltage regulator supplying power to the microcontroller, and an ADC voltage monitoring divider. The microcontroller runs a sampler task, a storage task, and an uploader task. Upon boot, the microcontroller checks a hardware reset reason register, and if it indicates a power brownout, it activates a write lockout mode that blocks the storage task from writing to the local storage interface for a predetermined stabilization window, while the sampler task continues to poll the 3-in-1 gas sensor array.

In its further embodiment, the 3-in-1 gas sensor array is a metal-oxide semiconductor package containing an Ammonia ($NH_3$) channel, a Reducing gas channel, and an Oxidizing gas channel on a single substrate.

In still its further embodiment, the microcontroller controls a heater enable GPIO pin connected to a heating element in the 3-in-1 gas sensor array, and the sampler task applies a warmup flag to the telemetry samples during an initial warm-up period after the heater enable GPIO pin is activated.

Furthermore, the charger controller IC regulates charging of the battery cell using a constant-current/constant-voltage charging cycle, and the ADC voltage monitoring divider comprises a high-impedance resistor network connected to a dedicated ADC pin of the microcontroller to measure the battery voltage without causing battery drain.

Additionally, during the write lockout mode, the microcontroller redirects the telemetry samples to a volatile RAM ring buffer, wherein the oldest telemetry sample in the volatile RAM ring buffer is overwritten when the buffer capacity is exceeded.

In a further embodiment, the Airquality edge device maintains a multi-tier fallback queue architecture comprising a live upload queue representing a first tier, a local storage queue representing a second tier, and a volatile RAM ring buffer representing a third tier, allowing data to fallback to RAM when the storage is write-locked.

In another embodiment, the local storage interface is configured under a short-filename FAT 8.3 standard, and the storage task writes the telemetry samples to daily-partitioned files named using a chronological pattern of exactly eight characters before the file extension.

In still another embodiment, the uploader task performs a chronological FIFO upload sequence by scanning the local storage interface for daily-partitioned files, sorting them alphabetically to isolate the oldest chronological file, reading from a byte offset stored in a persistent status file, streaming the data, and updating the byte offset upon successful transmission.

In yet another embodiment, upon reaching the end-of-file (EOF) of the selected file, the uploader task deletes the file if it corresponds to a historical day, and truncates the file back to its header length and resets the byte offset if it corresponds to the current active day.

According to another aspect of the invention, the microcontroller runs an embedded HTTP web server task configured to serve a local web page display rendering real-time gas readings, battery state of charge, and sensor calibration portals over a local Wi-Fi Access Point connection.

Furthermore, the remote telemetry server hosts a remote web client dashboard configured to display real-time dials and historical trend charts of the telemetry samples.

In still its further preferred embodiment, the Airquality edge device does not possess a synchronized system clock, and the sampler task generates a time-agnostic composite Universally Unique Identifier (UUID) for each telemetry sample at the moment of capture, comprising a unique hardware identifier of the edge device, a boot identifier comprising a random number generated once per boot cycle, and a sequential sample counter.

Additionally, the remote telemetry server receives the telemetry samples and deduplicates them by enforcing uniqueness on the composite UUID, permitting out-of-order and timestamp-corrected uploads without data replication.

Furthermore, the system includes a physical enclosure housing the microcontroller, the 3-in-1 gas sensor array, the local storage interface, and the portable power supply system, wherein the physical enclosure includes ventilation slots positioned above the 3-in-1 gas sensor array to facilitate gas diffusion, and an internal thermal barrier physically isolating the rechargeable battery cell and low-dropout voltage regulator from heat generated by the 3-in-1 gas sensor array.

In still its further preferred embodiment, the physical enclosure further includes an aperture aligning with the external USB charging port and light guide windows aligning with indicator LEDs of the microcontroller.

## Way(s) of implementation of the invention
The present invention provides for a portable resilient IoT environmental monitoring system. The system details and operations could be described thoroughly with the illustrations of FIG. 1 to FIG. 8 as follows below:

### 5.1 System Hardware Architecture (FIG. 1)
FIG. 1 illustrates the system hardware architecture of the **Airquality** edge device `100`. The edge device `100` comprises a microcontroller `102` (specifically an ESP32 dual-core system-on-chip with embedded SRAM and flash memory). The microcontroller `102` is electrically and logically coupled to:
*   A 3-in-1 metal-oxide semiconductor gas sensor array `104` (such as a MiCS-6814 package) connected to internal ADC channels.
*   A local storage interface `106` consisting of an SD card socket wired via a 1-bit or 4-bit bus interface (SDMMC or SPI) to a FAT-formatted SD card memory.
*   A wireless transceiver `108` configured to establish Wi-Fi STA and AP connections, as well as execute TCP/IP HTTP request clients.
*   A battery charging system `110` connected to provide power to the microcontroller `102` and receive charging power from an external source.

---

### 5.2 Software Component Task Isolation (FIG. 2)
FIG. 2 illustrates the software architecture task isolation running in the memory of the microcontroller `102`. The RTOS kernel schedules four primary tasks that execute concurrently:
1.  **Sampler Task `202`:** Periodically wakes up (e.g., every 5 seconds) to measure raw analog sensor voltages, execute ADC conversions, compute sensor resistance values, estimate concentration values, and package data into a structured telemetry sample.
2.  **Storage Task `204`:** Dequeues pending samples from local queues and appends them to CSV files on the local storage interface `106`.
3.  **Uploader Task `206`:** Manages network handshakes, device registration, and streams stored CSV records to the remote telemetry server.
4.  **Local Web Server Task `208`:** Serves an embedded HTTP server responsive to client browser requests over a local Wi-Fi Access Point connection.

---

### 5.3 Multi-Tier Fallback Queue Routing Algorithm (FIG. 3)
FIG. 3 illustrates the flowchart of the Multi-Tier Fallback Queue routing algorithm `300` executed by the Sampler Task `202` to control sample flow:
*   **Step 301 (Acquire Sample):** The Sampler Task `202` queries the sensor array `104` and battery divider `704` to create a `telemetry_sample_t` record.
*   **Step 302 (Online Check):** The system checks if Wi-Fi is connected and if the remote server is ready.
*   **Step 303 (Live Upload Route):** If the online check is successful, the sample is sent directly to a live upload queue (`s_live_upload_queue`). If enqueued successfully, the device remains in a `live` state, and the cycle exits.
*   **Step 304 (SD Card Check):** If offline or the live upload queue is full, the system checks if the local storage interface `106` is mounted and if the write lockout mode is active (`sd_card_write_allowed()`).
*   **Step 305 (Local Storage Enqueue):** If the SD card is mounted and write access is allowed, the sample is enqueued onto `s_sample_queue` for local writing.
*   **Step 306 (RAM Backlog Enqueue):** If the SD card is full, unmounted, or write-locked, the sample is routed to the volatile RAM backlog ring buffer (`s_ram_backlog`). If the buffer capacity is exceeded, the oldest sample is overwritten, preventing system memory leaks.

---

### 5.4 Power-Aware Write Lockout and Recovery (FIG. 4)
FIG. 4 illustrates the flowchart of the Power-Aware Write Lockout sequence `400` that guards the local storage interface `106` against voltage sags:
*   **Step 401 (Boot Reason Check):** Upon system initialization, the microcontroller `102` queries its reset reason register.
*   **Step 402 (Detect Brownout):** If the register indicates a brownout reset (`ESP_RST_BROWNOUT`), the system transitions to Step 403. Otherwise, it proceeds to Step 406.
*   **Step 403 (Initialize Lockout):** The system sets the `s_write_lockout` state flag to `true` and saves the current system tick count to `s_write_lockout_since_tick`.
*   **Step 404 (Verify Uptime):** For all write operations, the system compares the elapsed time since lockout against a threshold `SD_CARD_WRITE_LOCKOUT_RECOVERY_MS` (set to 2 minutes).
*   **Step 405 (Redirect Data):** If the elapsed time is less than 2 minutes, all write attempts to the local storage interface `106` are blocked, and samples are redirected to the RAM backlog.
*   **Step 406 (Clear Lockout):** If the elapsed time exceeds 2 minutes, the lockout flag is cleared, and write operations to the SD card are allowed to proceed.

---

### 5.5 Chronological FIFO Uploader and Storage Compaction (FIG. 5)
FIG. 5 illustrates the flowchart of the chronological FIFO uploader sequence `500` executed inside the Uploader Task `206`:
*   **Step 501 (Load State):** The uploader reads the current progress state from the state file `telupl.sta` on the local storage interface `106`.
*   **Step 502 (Select File):** If the state file has valid data, the specified file is selected. Otherwise, the uploader scans the SD card directory for files matching the pattern `tm*.csv` and sorts them alphabetically (which, due to the `tmYYMMDD` format, corresponds to oldest first).
*   **Step 503 (Stream Data):** The uploader opens the selected file, seeks to the saved byte offset, and streams the data chunk-by-chunk using HTTP POST requests to the remote telemetry server.
*   **Step 504 (Save Offset):** After each successful transmission, the uploader writes the updated byte offset to NVS or to `telupl.sta`.
*   **Step 505 (Check EOF):** Upon reaching the end-of-file (EOF), the uploader checks if the file is the current active daily log.
*   **Step 506 (Compaction):**
    *   If the file corresponds to a historical day, it is deleted from the disk (`unlink()`).
    *   If the file is the active daily log, the file is truncated back to the length of the CSV header, and the offset is reset, reclaiming storage space.

---

### 5.6 Time-Agnostic Deduplicating Composite UUID Structure (FIG. 6)
FIG. 6 illustrates the data structure of the composite UUID `602` and its representation in the SQL schema `604`. The composite UUID `602` is formatted as:
$$\text{UUID} = \text{DeviceID} + \text{"-"} + \text{BootID} + \text{"-"} + \text{SequenceNumber}$$
where `DeviceID` is derived from the MAC address of the wireless transceiver `108` (12 hex characters), `BootID` is a 16-hex character random number generated once per boot, and `SequenceNumber` is a 16-hex character incrementing index.

The remote database stores this in a normalized schema `604` comprising `telemetry_samples` (having `sample_uuid` as the Primary Key), `telemetry_locations`, `telemetry_gas_readings`, and `telemetry_heater_events` linked by foreign keys, enabling database deduplication.

---

### 5.7 Power Management and Battery Charging Circuits (FIG. 7)
FIG. 7 illustrates the electrical schematic of the battery charging system `110`. The system `110` comprises:
1.  **Rechargeable Battery Cell `701`:** A $3.7\text{ V}$ Lithium-Polymer (Li-Po) battery pack equipped with an internal protection circuit board (PCB).
2.  **CC/CV Charge Controller `702`:** A battery charger integrated circuit (e.g., TP4056 or MCP73831) connected to an external $5\text{ V}$ USB input `705`. The charger IC `702` operates in a Constant-Current (CC) phase to charge the cell `701` at a programmed rate, switching to a Constant-Voltage (CV) phase at $4.2\text{ V}$ to prevent overvoltage.
3.  **Low-Dropout (LDO) Regulator `703`:** A high-efficiency regulator (e.g., AP2112-3.3) that steps down the battery's variable voltage to a stable $3.3\text{ V}$ system rail ($V_{CC}$) `706` that powers the microcontroller `102`, local storage interface `106`, and gas sensors.
4.  **ADC Battery Monitoring Divider `704`:** A resistor divider comprising two high-resistance resistors ($R_1 = 100\text{ k}\Omega$ and $R_2 = 100\text{ k}\Omega$) connected between the positive battery terminal and ground, providing a scaled voltage ($V_{BAT\_SENSE} = V_{BAT} / 2$) directly to an ADC channel on the microcontroller `102`.

---

### 5.8 Decentralized Cryptographic Device Registration
In its further preferred embodiment, the system registers and authenticates devices dynamically. The microcontroller `102` derives its device ID from its MAC address. On boot, if no API token is stored in the NVS `device_reg` partition, the edge device sends a registration request containing its device ID and a signature:
$$\text{client\_hash} = \text{HMAC-SHA256}(\text{REGISTRATION\_SECRET}, \text{DeviceID})$$
The remote server verifies the hash, registers the device, and returns a unique, secure API token. The device caches this token in its NVS partition. Subsequent telemetry uploads include this unique API token in the HTTP Authorization headers (`Authorization: Bearer <api_token>`), preventing data spoofing.

---

### 5.9 Enclosure and Physical Housing Design (FIG. 8)
FIG. 8 illustrates the physical enclosure and ventilation housing `801` of the **Airquality** edge device. The physical device layout is protected by a compact portable enclosure `801` (such as a molded plastic or 3D printed housing). The enclosure `801` includes:
1.  **Diffusion Ventilation Slots `802`:** A series of grilles or intake apertures positioned on the housing exterior directly above the 3-in-1 gas sensor array `104` to allow ambient air to diffuse naturally across the sensor surface without requiring active fan power.
2.  **Internal Thermal Barrier `803`:** A physical dividing wall molded inside the enclosure `801` that isolates the battery cell `701` and LDO regulator `703` from the heat-producing gas sensor array `104`. This prevents the sensor heater from raising the temperature of the battery cell `701` (preventing fire hazard and thermal battery degradation).
3.  **USB Port Aperture `804`:** A cut-out aperture permitting external access to the USB charging input `705`.
4.  **LED Light Guide Apertures `805`:** Transparent acrylic light pipes or holes allowing indicator light from status LEDs (such as charger state or sensor warm-up alerts) to be visible on the external housing.

## C. ABSTRACT
A portable environmental monitoring system, designated as the Airquality platform, includes a battery-powered edge device `100` and a remote deduplicating server. The edge device `100` integrates a 3-in-1 gas sensor array `104` (measuring Ammonia, Reducing, and Oxidizing gases), a rechargeable battery `701`, a CC/CV charger IC `702`, an LDO voltage regulator `703`, and an ADC battery voltage monitor divider `704`, enclosed inside a housing `801` featuring ventilation slots `802` and an internal thermal barrier `803` protecting the battery. The device runs an RTOS partition separating a Sampler Task `202`, a Storage Task `204`, an Uploader Task `206`, and an embedded HTTP Web Server Task `208`. Upon a brownout reset detection, the edge device enters a temporary write lockout mode, protecting the local storage interface `106` from high-current write spikes while the voltage rails stabilize. Telemetry samples are routed via a multi-tier fallback queue (upload queue, local storage queue, or RAM ring buffer). Telemetry files are uploaded using a chronological FIFO stream, and are uniquely deduplicated at the server using a clock-independent composite UUID. Real-time data is presented to the user via local and remote graphical dashboards.
